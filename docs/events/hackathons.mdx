# Hackathons

## The Pressure Vessel

A hackathon is not a coding competition. It is a pressure vessel.

You have 24–72 hours. You have a team. You have a problem. You have no excuses.

We hack because building under constraint reveals what you actually know. It exposes the gaps between theory and practice. It forces you to make decisions with incomplete information — the defining skill of a Cybersecuritist.

## Our Philosophy

**Build to break. Break to learn. Learn to build better.**

Every hackathon we enter or host follows this cycle:

1. **Recon** — Understand the challenge. Read the rules. Scope the attack surface. Identify the scoring criteria.
2. **Architect** — Design the solution. Divide the work. Define interfaces. Choose the stack. Set up the repo, CI, deployment.
3. **Execute** — Code. Test. Iterate. Sleep in shifts. The clock does not stop.
4. **Hardening** — The last 20% of time is 80% of the result. Error handling. Edge cases. Documentation. Demo rehearsal.
5. **Post-Mortem** — Win or lose, we write the report. What worked. What failed. What we'd do differently with 2 more hours. The code is archived. The lessons are extracted.

## Types We Run

### Internal CTF Hackathons
Weekly. 4 hours. One challenge category per week — Web, Binary, Crypto, Forensics, Reversing, Misc. Scored. Ranked. The leaderboard is public.

### Research Sprints
Monthly. 48 hours. One research question — "Can we bypass PAC on ARM64?" "Is this new WAF bypass generalizable?" "Can we automate this vulnerability class?" Output: working exploit, detection rule, blog post.

### Tool-Building Weekends
Quarterly. 72 hours. A gap in our toolchain. "We need a better fuzzer for this protocol." "We need a visualization for this attack graph." Output: working tool, benchmarks, documentation.

### Community CTFs
We host. External teams invited. We design the challenges. We run the infrastructure. We mentor the participants. We learn by teaching.

## The Standard

No "participation trophies." No "good effort."

You either produced a working artifact or you didn't.

The artifact must:
- Compile/run without manual intervention
- Pass the test suite we wrote for it
- Include a README that a stranger can follow
- Have a post-mortem document (what broke, what held, what you'd change)

If it doesn't meet the standard, it doesn't ship. You iterate until it does.

## Infrastructure

We build our own platform. CTFd customized. Scoreboard real-time. Challenge deployment automated. Container orchestration with Kubernetes. Monitoring with Prometheus/Grafana. Logging with Loki. All built and maintained by members.

## The Output

Every hackathon produces:
- **Code** — Open sourced. Documented. Benchmarked.
- **Writeups** — Technical depth. Reproducible. Educational.
- **Tools** — Added to the club arsenal. Maintained.
- **People** — Sharper. More resilient. Better at making decisions under pressure.
